## Key Decisions Made

1. **Convex Deployment Topology**: Established four-tier pattern:
   - **Dev instances** (per-developer, `VITE_CONVEX_URL` in `.env.local`)
   - **Staging** (optional long-lived preview for testing)
   - **Production** (deployed by main-branch push via deploy-main.yml)
   - **CI/test-only Convex instance** (ephemeral, for PR validation)
   
   *Rationale:* Convex has no separate "Development" tier; preview deployments serve the staging function. This topology provides clear separation of concerns and allows PRs to validate without affecting production.

2. **Workflow Split**: Separated CI (validation) from deployment (production push):
   - `.github/workflows/ci.yml` — runs on every push and PR; validates but does not deploy
   - `.github/workflows/deploy-main.yml` — runs only on main pushes; deploys to production
   
   *Rationale:* PRs should validate without side effects. Production deployment is only on merged main.

3. **Documentation as Canonical Truth**: Placed Convex-topology explanation in CLAUDE.md (project-wide maintainer guide) rather than buried in workflow comments.
   
   *Rationale:* Future PRs touching Convex will read CLAUDE.md first; this prevents topology misunderstandings.

## Code Changes Summary

### `.github/workflows/deploy-main.yml` (new)
```yaml
name: Deploy to Convex Production
on:
  push:
    branches:
      - main
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
      - run: pnpm install
      - run: pnpm convex deploy --prod --url "${{ secrets.CONVEX_DEPLOY_KEY }}"
```
